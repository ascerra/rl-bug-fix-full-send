The Problem

Slow Turnaround

Bug reports sit in backlogs for days. By the time a fix ships, the reporter has moved on or worked around it.

Context Switching

Engineers interrupt deep work to triage. They read the issue, clone the repo, reproduce the bug, context-switch into the codebase — all before writing a single line of fix.

Review Bottleneck

Fixes wait hours or days for code review. Reviewers lack context on the bug, leading to shallow reviews that miss subtle issues.

What if we could go from bug report to reviewed PR in minutes, not days?

The Ralph Loop Concept

An agentic iteration pattern applied at two levels

Meta Loop

Builds and improves the engine itself.

  • 51 completed iterations
  • Each run adds capability or fixes a deficiency
  • Principles extracted automatically from patterns
  • Compound improvement over time

Production Loop

Fixes bugs in real GitHub repositories.

  • Runs in GitHub Actions — no local setup
  • Targets any repo with any language stack
  • Full triage → fix → review → PR pipeline
  • Escalates to humans when stuck

Phase Pipeline

5 phases, each running a full OODA cycle independently

Triage

Classify severity, find affected files, attempt reproduction, detect repo stack

Implement

Root cause analysis, minimal fix, run linters, inner iteration loop

Review

Zero-trust independent review, correctness + security + path consistency

Validate

Verify minimal diff, generate PR description, create PR, monitor CI

Report

Zero Trust Between Phases

The review phase re-reads the issue and diff from scratch. It does not trust the implementation phase's summary.

Escalation, Not Hallucination

When stuck after max retries, the engine escalates to a human with full context — never guesses or ships a bad fix.

Testing Strategy: CI-First

Test execution is disabled by default — this is deliberate

Why Not Run Tests?

  • Target repos need runtimes we don't control
  • Deps may require Docker, databases, kind clusters
  • Test suites can run 20+ minutes
  • Flaky tests waste the iteration budget
  • Running arbitrary shell = security surface

What We Do Instead

  • Linting is always on (fast, safe, universal)
  • The PR triggers the repo's own CI pipeline
  • Repo CI has the correct matrix, services, secrets
  • Repos can opt in via .rl-config.yaml
  • Three modes: disabled → opportunistic → required

Why this matters

The target repo's CI is the authoritative test environment — it has the build matrix, service containers, and secrets the engine never will. Delegating validation to CI is more reliable than attempting to replicate it.

Key Takeaways

Agentic Iteration Works

Feed failures back as context. The engine improves on each pass, converging on correct fixes without human intervention.
